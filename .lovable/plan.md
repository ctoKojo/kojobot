

# خطة التوافق مع البيانات القديمة - النظام المالي

## المشكلة
عندك طلاب أوريدي موجودين في السيستم ودافعين (3 شهور أو شهر). لما ننزل النظام المالي الجديد، لازم يكون فيه طريقة سهلة تدخل بيها بيانات الطلاب القدام من غير مشاكل.

## الحل: شاشة "تسجيل اشتراك يدوي" للمدير

### الفكرة
المدير يقدر يفتح بروفايل أي طالب موجود ويعمله اشتراك جديد مع تحديد:
- **خطة التسعير** (الباقة)
- **نوع الدفع** (كامل 3 شهور / تقسيط شهري)
- **تاريخ البداية** (ممكن يكون في الماضي - مثلاً الطالب بدأ من شهرين)
- **المبلغ المدفوع فعلاً** (اللي اتدفع لحد دلوقتي)

### سيناريو عملي

**مثال 1: طالب دافع 3 شهور كامل**
```text
الباقة: Kojo Squad Offline
نوع الدفع: كامل (3 شهور)
تاريخ البداية: 2026-01-01
تاريخ النهاية: 2026-04-01 (أوتوماتيك)
المبلغ الإجمالي: 2500
المدفوع: 2500
المتبقي: 0
الحالة: فعال - مدفوع بالكامل
```

**مثال 2: طالب دافع شهر واحد (تقسيط)**
```text
الباقة: Kojo Squad Offline
نوع الدفع: تقسيط شهري
تاريخ البداية: 2026-01-15
تاريخ النهاية: 2026-04-15 (أوتوماتيك)
المبلغ الإجمالي: 2500
القسط الشهري: 1000
المدفوع: 1000 (أول شهر)
المتبقي: 1500
موعد الدفع القادم: 2026-02-15
```

**مثال 3: طالب دافع شهرين من 3**
```text
الباقة: Kojo Core Offline
نوع الدفع: تقسيط شهري
تاريخ البداية: 2025-12-10
تاريخ النهاية: 2026-03-10 (أوتوماتيك)
المبلغ الإجمالي: 4500
القسط الشهري: 1800
المدفوع: 3600 (شهرين)
المتبقي: 900
موعد الدفع القادم: 2026-02-10
```

## التصميم التقني

### 1. جدول `pricing_plans` - بيانات أولية
عند إنشاء الجدول، يتم إدخال التسعيرة الحالية تلقائياً:

| الباقة | الموقع | السعر الأصلي | الخصم | سعر 3 شهور | سعر شهري |
|--------|--------|-------------|-------|-----------|----------|
| Kojo Squad | Offline | 4000 | 40% | 2500 | 1000 |
| Kojo Core | Offline | 6500 | 30% | 4500 | 1800 |
| Kojo X | Offline | 7500 | 35% | 5000 | 2200 |
| Kojo Squad | Online | 3750 | 45% | 2000 | 850 |
| Kojo X | Online | 4200 | 50% | 2100 | 1400 |

### 2. تعديل جدول `subscriptions`
إضافة أعمدة جديدة مع الحفاظ على التوافق:

- `pricing_plan_id` (nullable) - لو فاضي يعني اشتراك قديم بدون ربط
- `payment_type` (default: 'full') - كامل أو تقسيط
- `installment_amount` (nullable) - قيمة القسط
- `next_payment_date` (nullable) - موعد الدفع القادم
- `is_suspended` (default: false) - هل الحساب موقوف

كل الأعمدة الجديدة nullable أو لها default، فالبيانات القديمة (لو كان فيه) مش هتتأثر.

### 3. جدول `payments` - سجل المدفوعات
عند تسجيل اشتراك قديم، المدير يحدد المبلغ المدفوع والنظام يسجل دفعة واحدة بقيمة المبلغ ده كـ "دفعة مسبقة" (prior_payment).

### 4. شاشة تسجيل الاشتراك (في بروفايل الطالب)

الخطوات:
1. المدير يفتح بروفايل الطالب ويضغط "إنشاء اشتراك"
2. يختار الباقة من dropdown (تتفلتر أوتوماتيك حسب نوع المجموعة وطريقة الحضور)
3. يختار نوع الدفع (كامل / تقسيط)
4. يحدد تاريخ البداية (يقدر يختار تاريخ في الماضي)
5. النظام يحسب أوتوماتيك: تاريخ النهاية، المبلغ الإجمالي، القسط
6. المدير يدخل المبلغ المدفوع فعلاً
7. النظام يحسب المتبقي وموعد الدفع القادم

### 5. منطق حساب موعد الدفع القادم
```text
لو الدفع كامل:
  next_payment_date = NULL (مفيش أقساط)

لو تقسيط:
  عدد الأقساط المدفوعة = المدفوع / قيمة القسط (تقريب لأسفل)
  next_payment_date = تاريخ_البداية + (عدد_الأقساط_المدفوعة * 30 يوم)
  
  لو next_payment_date < النهارده → الطالب متأخر (يتوقف حسابه)
  لو next_payment_date >= النهارده → الطالب في الأمان
```

### 6. نظام القفل التلقائي
- Edge Function `check-payment-dues` تفحص الاشتراكات يومياً
- لو `next_payment_date` عدى وفيه مبلغ متبقي → `is_suspended = true`
- الطالب يشوف صفحة "حسابك موقوف" بدل الداشبورد
- لما المدير يسجل دفعة جديدة → `is_suspended = false` أوتوماتيك + يتحدث `next_payment_date`
